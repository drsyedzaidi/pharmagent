Lesson 01: The Tidyverse: Tibbles, Pipes, and the Tidy Data Standard

Objective. Explain what the tidyverse is and why it exists, use tibbles instead of base data frames with an understanding of what changed, compose pipelines with the pipe operator, and apply Wickham's tidy-data definition to classify a dataset as tidy or messy.

Prerequisites. Base R: vectors, data frames, functions, indexing.

Concept (from first principles)

What the tidyverse actually is

The tidyverse is not one package — it is a family of ~25 R packages (ggplot2, dplyr,
tidyr, readr, purrr, tibble, stringr, forcats, and more) that share three
design commitments:

  1. A common data structure. Every function that can take tabular input expects (and
    returns) a tibble, not a base data.frame.
  2. A common grammar. Verbs are functions: filter() selects rows, select() selects
    columns, mutate() creates columns, summarise() collapses rows. The first argument
    is always the data, which is what makes piping work.
  3. A common philosophy on tidy data. Every package assumes the input is already
    "tidy" (defined below) — messiness is handled explicitly with tidyr, not silently
    inside plotting or modeling code.

library(tidyverse) loads the core eight packages at once. You will use readr and
readxl in this lesson and lesson 2; dplyr/tidyr start in Phase 02.

Tibbles vs base data.frame

A tibble (tbl_df) is a data frame with stricter, more predictable behavior. Four
concrete differences:

Behavior base data.frame tibble
Partial column-name matching df$se` matches `subject_id` if unambiguous — silent | `tb$se returns NULL with a warning-free NULL, forcing you to spell it out
String columns stringsAsFactors converted strings to factors by default pre-R-4.0 Strings always stay character; never auto-factored
Printing Prints all rows/columns, can flood the console Prints first 10 rows and only the columns that fit the width, plus each column's type
List-columns Awkward, error-prone to create and print First-class: a tibble column can hold a list (e.g. nested model fits, vectors of varying length per row) 
library(tibble)

# Base data.frame: partial matching silently "works"
df <- data.frame(subject_id = 1:3, concentration = c(1.2, NA, 3.4))
df$conc        # partial match on "concentration" — returns the column, no warning

# Tibble: no partial matching
tb <- as_tibble(df)
tb$conc        # returns NULL — forces exact names, catches typos early

# Printing: tibble shows type + truncates, base data.frame does not
print(tb)
#> # A tibble: 3 x 2
#>   subject_id concentration
#>        <int>         <dbl>
#> 1          1           1.2
#> 2          2          NA
#> 3          3           3.4

Row names are also treated differently: tibbles do not support rownames as a parallel
identifier — if you need a subject ID, it should be an explicit column, which is itself
a tidy-data discipline (see below).

The pipe: |> and %>%

Base R (4.1+) ships a native pipe, |>. The tidyverse historically used magrittr's
%>%. Both take the expression on the left and insert it as the first argument of the
function on the right:

x |> f()          # equivalent to f(x)
x |> f(y)         # equivalent to f(x, y)
x %>% f(y)        # magrittr — same result, older syntax

Why pipe instead of nesting function calls? Compare:

# Nested — read inside-out, hard to trace data flow
round(mean(sqrt(abs(c(-4, 9, 16)))), 2)

# Piped — read top-to-bottom / left-to-right, matches the order of operations
c(-4, 9, 16) |>
  abs() |>
  sqrt() |>
  mean() |>
  round(2)

The piped version reads like a sentence: "take this vector, then take absolute value,
then square root, then mean, then round." This is the single biggest reason tidyverse
code is more readable than equivalent base R for multi-step transformations.

|> vs %>% — practical differences:
- |> is built into R itself (no package needed) since R 4.1.0.
- %>% supports a placeholder (.) for inserting the piped value in a non-first
argument position, e.g. df %>% lm(y ~ x, data = .). Base |> gained a similar _
placeholder in R 4.2+ but requires naming the argument: df |> lm(y ~ x, data = _).
- New tidyverse code and this course default to |>; you will still see %>% constantly
in existing scripts, papers, and Stack Overflow answers — read both fluently.

Tidy data: Wickham's definition

A dataset is tidy when three conditions hold simultaneously:

  1. Each variable forms a column.
  2. Each observation forms a row.
  3. Each type of observational unit forms a table.

This sounds abstract until you see the violation. Consider concentration-time data
recorded wide, one column per nominal timepoint — a common way labs export data, and a
common way it arrives messy:

# WIDE (messy for analysis): one row per subject, one column per timepoint
wide <- tibble::tribble(
  ~subject_id, ~t0,  ~t1,  ~t2,  ~t4,
  "S01",        0.0, 4.2,  3.1,  1.4,
  "S02",        0.0, 3.9,  2.8,  1.1
)
wide

Here t0, t1, t2, t4 are not four variables — they are four values of one
variable (time), each paired with one value of another variable (conc). The column
headers are data, not variable names. This violates rule 1.

# LONG (tidy): one row per observation (subject x time)
long <- tibble::tribble(
  ~subject_id, ~time, ~conc,
  "S01",        0,    0.0,
  "S01",        1,    4.2,
  "S01",        2,    3.1,
  "S01",        4,    1.4,
  "S02",        0,    0.0,
  "S02",        1,    3.9,
  "S02",        2,    2.8,
  "S02",        4,    1.1
)
long

Now subject_id, time, and conc are each a column, and each row is exactly one
measurement (one subject at one time). ggplot2::geom_line(aes(time, conc, group = subject_id)) works directly on the long form; it cannot consume the wide form at all
without reshaping first. Reshaping wide ↔ long is tidyr::pivot_longer() /
pivot_wider() — covered in Phase 02 — but recognizing the violation is a Phase 01
skill: you must be able to look at any table and say "this is wide" or "this is tidy"
before you write a single line of cleaning code.

The third rule — one table per observational unit — matters for relational data: subject
demographics (one row per subject) and dosing records (one row per dose event) are two
different observational units and belong in two different tables, joined later on
subject_id (Phase 03).

Worked Example

Scenario. A lab hands you a spreadsheet of steady-state trough concentrations for
three subjects, recorded across three visits, in wide format. Diagnose the tidy-data
violation and reshape by hand (conceptually) before any Phase-02 tooling exists.

library(tibble)

trough_wide <- tribble(
  ~subject_id, ~sex, ~visit1, ~visit2, ~visit3,
  "S01",        "M",  8.2,     7.9,     8.5,
  "S02",        "F",  6.1,     6.4,     5.9,
  "S03",        "M",  9.0,     8.8,     9.3
)
trough_wide

Step 1 — Identify the variables actually present. There are four conceptual
variables: subject_id, sex, visit number, and trough concentration. The wide
table only gives two of them explicit columns (subject_id, sex); visit1/visit2/
visit3 smuggle "visit number" into column names, and the cell values are the
concentrations.

Step 2 — Predict the tidy shape. Every combination of subject and visit is one
observation, so the tidy table needs one row per subject-visit pair:

trough_long <- tribble(
  ~subject_id, ~sex, ~visit, ~trough,
  "S01", "M", 1, 8.2,
  "S01", "M", 2, 7.9,
  "S01", "M", 3, 8.5,
  "S02", "F", 1, 6.1,
  "S02", "F", 2, 6.4,
  "S02", "F", 3, 5.9,
  "S03", "M", 1, 9.0,
  "S03", "M", 2, 8.8,
  "S03", "M", 3, 9.3
)
trough_long

Step 3 — Confirm against the three rules. subject_id, sex, visit, trough are
each one column ✓. Each row is exactly one subject observed at one visit ✓. One
observational unit (a trough measurement) per table ✓. This table is tidy and is now
ggplot2- and dplyr-ready; the wide version was not.

Common Pitfalls

  1. Assuming data.frame() and tibble() behave identically. Code that relies on
    partial column matching (df$conc matching concentration) will silently break — or
    worse, silently return NULL — the moment the object becomes a tibble. Always spell
    out full column names.

  2. Treating wide format as "just how the data comes" and building dplyr/ggplot2
    code around it.     Every downstream tidyverse tool assumes long/tidy input. Fighting
    wide data with ad hoc column-by-column code is more work than reshaping once, up
    front (Phase 02).

  3. Mixing %>% and |> inconsistently within one script. Both work, but pick one
    per script/project for readability; do not alternate line to line.

  4. Forgetting the placeholder syntax differs. df %>% lm(y ~ x, data = .) uses .;
    the base pipe needs a named placeholder, df |> lm(y ~ x, data = _), and only works
    inside a function call with a named argument — x |> f(_, 2) for positional
    placement is not supported.

  5. Believing "tidy" means "clean." A dataset can be perfectly tidy (each variable a
    column, each observation a row) and still contain typos, wrong types, duplicates, or
    missing-value sentinels. Tidy is a shape property; clean is a content property.
    Lesson 03 covers the content audit.

Code

lesson_01.R

# Lesson 01: The Tidyverse — Tibbles, Pipes, and the Tidy Data Standard
# Self-contained script. Run top to bottom.

# install.packages("tidyverse")  # run once if not already installed
library(tidyverse)   # attaches ggplot2, dplyr, tidyr, readr, purrr, tibble, stringr, forcats
library(tibble)

set.seed(1)  # only needed for the randomized ID example below

# ---------------------------------------------------------------------------
# 1. Which core packages did we just attach?
# ---------------------------------------------------------------------------
tidyverse_packages()   # lists the packages considered "core tidyverse"

# ---------------------------------------------------------------------------
# 2. Base data.frame vs tibble: four concrete behavioral differences
# ---------------------------------------------------------------------------

# 2a. Partial column-name matching
df <- data.frame(subject_id = 1:3, concentration = c(1.2, NA, 3.4))
tb <- as_tibble(df)

cat("Base data.frame partial match df$conc  ->", df$conc, "\n")   # silently works
cat("Tibble partial match tb$conc           ->",
    if (is.null(tb$conc)) "NULL (no partial match)" else tb$conc, "\n")

# 2b. stringsAsFactors behavior
df_str <- data.frame(sex = c("M", "F", "M"))
tb_str <- tibble(sex = c("M", "F", "M"))
cat("Base data.frame sex column class:", class(df_str$sex), "\n")  # character (R >= 4.0)
cat("Tibble sex column class:         ", class(tb_str$sex), "\n")  # always character

# 2c. Printing: tibble truncates and shows column types; data.frame dumps everything
big_df <- data.frame(id = 1:20, val = rnorm(20))
big_tb <- as_tibble(big_df)
print(big_tb)   # shows "# A tibble: 20 x 2" + first 10 rows + column types

# 2d. List-columns: first-class in tibbles, awkward in base data.frame
tb_list_col <- tibble(
  subject_id = c("S01", "S02"),
  visit_times = list(c(0, 1, 2, 4), c(0, 2, 4, 8, 24))  # unequal-length vectors per row
)
print(tb_list_col)
cat("visit_times for S02:", tb_list_col$visit_times[[2]], "\n")

# ---------------------------------------------------------------------------
# 3. Piping: |> (base) vs %>% (magrittr) vs nested calls
# ---------------------------------------------------------------------------
raw_vals <- c(-4, 9, 16)

# Nested (inside-out, hard to read)
nested_result <- round(mean(sqrt(abs(raw_vals))), 2)

# Base pipe (left to right)
piped_result <- raw_vals |>
  abs() |>
  sqrt() |>
  mean() |>
  round(2)

# magrittr pipe — identical result, older syntax you will see everywhere
piped_result_magrittr <- raw_vals %>%
  abs() %>%
  sqrt() %>%
  mean() %>%
  round(2)

stopifnot(nested_result == piped_result, piped_result == piped_result_magrittr)
cat("Nested:", nested_result, "| Base pipe:", piped_result,
    "| magrittr pipe:", piped_result_magrittr, "\n")

# Placeholder syntax difference: magrittr `.` vs base R `_`
mini <- tibble(x = 1:5, y = c(2, 4, 5, 4, 5))

fit_magrittr <- mini %>% lm(y ~ x, data = .)          # `.` = piped data, any position
fit_base     <- mini |> lm(y ~ x, data = _)            # `_` = piped data, must be named arg
stopifnot(identical(coef(fit_magrittr), coef(fit_base)))
cat("Both pipes produced identical lm() fits.\n")

# ---------------------------------------------------------------------------
# 4. Wide -> long by hand (tribble only; pivot_longer() arrives in Phase 02)
# ---------------------------------------------------------------------------
trough_wide <- tribble(
  ~subject_id, ~sex, ~visit1, ~visit2, ~visit3,
  "S01",        "M",  8.2,     7.9,     8.5,
  "S02",        "F",  6.1,     6.4,     5.9,
  "S03",        "M",  9.0,     8.8,     9.3
)
cat("\nWide (messy shape) trough data:\n")
print(trough_wide)

trough_long <- tribble(
  ~subject_id, ~sex, ~visit, ~trough,
  "S01", "M", 1, 8.2,
  "S01", "M", 2, 7.9,
  "S01", "M", 3, 8.5,
  "S02", "F", 1, 6.1,
  "S02", "F", 2, 6.4,
  "S02", "F", 3, 5.9,
  "S03", "M", 1, 9.0,
  "S03", "M", 2, 8.8,
  "S03", "M", 3, 9.3
)
cat("\nLong (tidy shape) trough data:\n")
print(trough_long)

# ---------------------------------------------------------------------------
# 5. Tidy/messy classification exercise
# ---------------------------------------------------------------------------
# Table A: tidy (one row per subject-time observation)
table_a <- tribble(
  ~subject_id, ~time, ~conc,
  "S01", 0, 0.0,
  "S01", 1, 4.2
)

# Table B: messy — column headers are values (dose levels), not variable names
table_b <- tribble(
  ~subject_id, ~dose_50mg, ~dose_100mg, ~dose_200mg,
  "S01",        1.1,        2.4,         5.0
)

# Table C: messy — two variables (subject demographics + dose events) crammed in one table
table_c <- tribble(
  ~subject_id, ~age, ~sex, ~dose_time, ~dose_amt,
  "S01",        34,   "M",  0,          100,
  "S01",        34,   "M",  12,         100
)

classify_tidy <- function(name, is_tidy, reason) {
  cat(sprintf("%-10s %-7s %s\n", name, if (is_tidy) "TIDY" else "MESSY", reason))
}
cat("\nClassification:\n")
classify_tidy("table_a", TRUE,  "each variable a column, each row one observation")
classify_tidy("table_b", FALSE, "dose_50mg/100mg/200mg are values of 'dose', not variables")
classify_tidy("table_c", FALSE, "mixes two observational units (subject, dose event) in one table")

cat("\nLesson 01 complete.\n")

Check

You run df$conc on a tibble whose only concentration column is named 'concentration', expecting the partial matching you're used to from base data.frame. What happens?
  • A. It returns the concentration column, with a warning about partial matching
  • B. It returns NULL — tibbles do not support partial column-name matching
  • C. It throws a hard error and stops execution
  • D. It returns the first column of the tibble regardless of name

Answer: B

Tibbles deliberately disable partial matching on $ to prevent silent bugs from typos or abbreviated names. df$conc on a base data.frame would silently match 'concentration', but the identical code on a tibble returns NULL with no error, forcing you to spell out the full column name.

A dataset has one row per patient and separate columns dose_week1, dose_week2, dose_week3. Is this tidy data by Wickham's definition?
  • A. Yes — each column is clearly labeled and each patient has one row
  • B. Yes, as long as all three columns are numeric
  • C. No — 'week' is a variable whose values (week1/2/3) have been spread into column names instead of a single column
  • D. No — tidy data requires every column to be numeric

Answer: C

This is the classic wide-format violation of tidy data rule 1 (each variable forms a column). dose_week1/2/3 are not three variables — they are three values of one variable ('week'), each paired with one value of another variable (dose). The tidy form would have columns patient_id, week, dose with one row per patient-week observation.

What is the key syntactic difference between the magrittr pipe placeholder (.) and the base R pipe placeholder (_) when inserting piped data into a non-first argument?
  • A. There is no difference — both work identically in any position
  • B. magrittr's `.` can be used positionally anywhere; base R's `_` must be used as a named argument (e.g. data = _)
  • C. Base R's `_` works in any position; magrittr's `.` must be a named argument
  • D. Both require the argument to be named and cannot be used positionally

Answer: B

magrittr's `.` placeholder is flexible — df %>% lm(y ~ x, data = .) works because `.` stands in for the piped object wherever it appears. Base R's `|>` placeholder `_` (R >= 4.2) is stricter: it must be attached to a named argument, so df |> lm(y ~ x, data = _) works but a bare positional `_` does not.

Tidyverse vs Base R Cheat Sheet

Tibble vs data.frame Behavior

Behavior base data.frame tibble
Partial column matching (df$conc for concentration) Matches silently Returns NULL, no match
String columns Historically auto-factored (stringsAsFactors, pre-4.0 default) Always stays character
Printing Dumps all rows/columns Shows first 10 rows, fits console width, prints column types
List-columns Awkward, error-prone First-class citizen
Row names Can carry a parallel rowname identifier No rowname identifier — use an explicit ID column
Creation data.frame(...) tibble(...) or as_tibble(df)
Row-wise literal construction Manual data.frame(..., stringsAsFactors = FALSE) tribble(~col1, ~col2, val, val, ...) — readable, row-oriented

Pipe Syntax Reference

Feature %>% (magrittr) |> (base R, 4.1+)
Requires package magrittr / tidyverse None — built into R
Basic form x %>% f(y) = f(x, y) x |> f(y) = f(x, y)
Placeholder for non-first arg . — usable anywhere _ — must be a named argument (R >= 4.2)
Example df %>% lm(y ~ x, data = .) df \|> lm(y ~ x, data = _)
Performance Slight function-call overhead Syntax transform, effectively free
Where you'll see it Nearly all existing tidyverse code, papers, Stack Overflow New code, this course's default

Rule of thumb: pick one pipe per script and stay consistent. Read both fluently —
you cannot avoid %>% in the wild.

Tidy Data — The Three Rules (Wickham 2014)

  1. Each variable forms a column.
  2. Each observation forms a row.
  3. Each type of observational unit forms a table.

Wide (messy) vs Long (tidy) — Quick Visual

S01         0.0  4.2  3.1              S01         0     0.0

Column headers that are values (timepoints, dose levels, visit numbers) — not
variable names — are the single most common tidy-data violation. Reshaping tools
(pivot_longer()/pivot_wider()) arrive in Phase 02; for now, the skill is
recognizing the violation by eye.

Tidy Is Not the Same as Clean

Property What it means Checked by
Tidy Correct shape: one variable per column, one observation per row Eyeballing structure (this lesson)
Clean Correct content: right types, no duplicates, no sentinel codes, consistent categories Systematic audit (Lesson 03)

A dataset can be tidy and still be full of "BLQ" strings in a numeric column, -99
sentinel missing codes, or duplicate subject rows.
