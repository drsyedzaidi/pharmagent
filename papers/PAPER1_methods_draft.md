## Methods

### Framework overview

The framework lets a single analytical agent fit one or more candidate population pharmacokinetic (PK) models across several estimation engines and then select a preferred model on a footing that is valid across those engines. It is organised around one normalised result object and a small adapter contract, so that a new engine can be added without altering the comparison or selection logic. Every engine's fit is reduced to a common `EngineResult` record; a runner fans a set of candidate models across the available engines; and a selection step ranks the resulting records. The comparison is exposed to the agent as an audited tool and, at a higher level, as a human-gated workflow (described below).

A candidate model is specified by a `CandidateSpec`: a structural model key drawn from the platform's model registry, the parameters carrying between-subject variability, the residual-error model, and, for the native engine, the estimation method. The dataset is prepared once into a single subject contract — a list of per-subject records (dosing history, observation times and concentrations, and weight) — which each adapter consumes; an external engine may then serialise that shared contract into its own native input format (for example, the nlmixr2 adapter writes a NONMEM-style CSV that R reads), but the contract itself is built only once.

### Normalised engine result

Each fit is reduced to an `EngineResult` with three groups of fields. The first is provenance: the engine identifier and version, the model name, a convergence flag, and wall-clock runtime. The second is the native likelihood — an optional objective function value (OFV) and, where they can be defined, the Akaike and Bayesian information criteria (AIC, BIC). These fields are retained but are treated as *within-engine only* for reasons given below. The native fitter returns an OFV but not AIC or BIC, so the framework derives them as OFV + 2k and OFV + k·ln(n), where k is the estimated-parameter count (structural parameters, covariate coefficients, between-subject variance terms, and residual-error terms) and n is the number of observations. An external adapter may populate only the OFV field and leave AIC/BIC unset when its parameter count is not exposed (the nlmixr2 adapter reports OFV alone); this is immaterial to selection, which never reads these fields. The third group holds the engine-agnostic goodness fields on which selection depends: the log-scale prediction root-mean-square error, the log-scale coefficient of determination, the mean log prediction bias, and a prediction-corrected visual-predictive-check coverage. A failed or unavailable engine yields a record whose status marks it as such rather than raising an exception, so that one engine cannot abort the comparison.

### Estimation engines and the adapter contract

Every engine implements the same adapter contract: an availability check and a `fit` method that maps a candidate specification and a subject list to an `EngineResult`. Three adapters are provided.

The native adapter wraps the platform's own first-order conditional estimation with interaction (FOCE-I) and stochastic-approximation expectation-maximisation (SAEM) routines. FOCE-I and SAEM are surfaced as two distinct engines, which is deliberate: it is exactly the pairing for which native objective functions are not comparable and for which prediction-based ranking is therefore required. `[CITATION NEEDED: FOCE/Laplace — e.g. Wang 2007]` `[CITATION NEEDED: SAEM — Kuhn & Lavielle 2005]`

The nlmixr2 adapter is a live external engine that runs in R. For a supported structural model it generates the corresponding nlmixr2 model, writes the subject list to a NONMEM-style dataset (identifier, time, dependent variable, amount, event, and compartment columns), invokes R as a subprocess, and parses the returned population estimates (fixed effects, between-subject coefficients of variation, and the proportional residual term) from a JSON payload. Allometric weight scaling is injected into the generated model so that the external estimates are weight-centred in the same way as the native fit, which is necessary for the two engines to be scored on identical predictions. The adapter reports itself available only when R, the nlmixr2 package, and a JSON library are all present, so the framework runs unchanged on systems without R. In the present version the adapter supports one-compartment oral and intravenous models; other structures return a failed record. `[CITATION NEEDED: nlmixr2 — Fidler et al.]`

A deterministic mock adapter is also provided. It runs the native fit once and perturbs the parameters by a fixed multiplicative factor, which produces reproducible `EngineResult` records for a plausibly better or worse engine without any external dependency, and is used for testing and demonstration.

### Rationale for engine-agnostic ranking

An estimation engine's objective function is defined on its own algorithm's likelihood approximation, so its magnitude is not commensurable with another engine's. Comparing OFV, AIC, or BIC across engines is therefore not a valid basis for selecting a model, even though it is valid within a single engine. The framework enforces this distinction structurally: the selection routine's ranking key cannot read the native-likelihood fields, and the within-engine likelihoods are reported in a separate table that is bucketed by engine so that a cross-engine likelihood comparison is not representable. Cross-engine ranking uses only prediction metrics computed by a single shared predictor, as follows.

### Engine-agnostic prediction scoring

Every engine is scored by a common procedure that depends only on its population parameters, so that the sole difference between engines at scoring time is the parameter values and never the prediction machinery. Given an engine's typical parameters, between-subject variances, and residual-error terms, an empirical-Bayes (maximum a posteriori) estimate of each subject's random effects is computed with the platform's own conditional-estimation step; the resulting individual parameters generate individual predictions, and the typical parameters generate population predictions. From these, the framework computes the log-scale prediction root-mean-square error and coefficient of determination against the observed concentrations, the mean log prediction bias, and a prediction-corrected visual-predictive-check coverage defined as the fraction of time bins in which the observed median falls within the 90% confidence interval of the simulated median. `[CITATION NEEDED: pcVPC — Bergstrand et al. 2011]` Recomputing individual predictions on one side, rather than using each engine's own post-hoc estimates, standardises the comparison; its limitation — that an engine's native structural predictor is not exercised — is considered in the Discussion.

### Winner selection

The runner collects one `EngineResult` per candidate and engine. Selection considers only converged records with an acceptable status and ranks them by prediction accuracy: ascending prediction root-mean-square error first, with ties broken by higher visual-predictive-check coverage, then higher coefficient of determination, then lower absolute bias. The native-likelihood fields do not enter the ranking key. The selection output additionally carries the per-engine likelihood table and an explicit note that objective functions are not compared across engines, so that a reader of the result cannot mistake the basis of the decision.

### Agent integration and audited workflow

The comparison is registered as a modeler tool that the analytical agent can invoke, and as a corresponding programmatic endpoint. Each invocation is wrapped by an append-only, hash-chained audit record; the engine results are reduced to a serialisable form for hashing. At the workflow level the comparison is embedded in a fit-then-confirm pipeline: a structural model is selected, the cross-engine comparison is run, and an independent adversarial-review step is presented to the human operator as a decision gate before any downstream use. The review inspects the structural fit and the cross-engine result and raises calibrated findings — for example, that the winning model was confirmed by only one engine — so that the pipeline surfaces the limits of its own evidence rather than obscuring them.

### Software verification

The implementation was verified in two complementary ways. A deterministic test suite exercises the schema, the runner's graceful handling of absent and failed engines, the selection invariants (in particular, that an engine with a lower objective function but poorer predictions does not win), and the external adapter against a live fit; 28 tests specific to the framework pass, together with additional coverage of the modeling-review gate in the adversarial-review suite. In addition, the implementation was subjected to repeated adversarial review, in which independent automated reviewers attempted to refute the correctness of the scoring, selection, and engine adapters, and each surfaced finding was verified against the source before being accepted. This process identified and led to the correction of a series of defects — among them an asymmetry in allometric scaling between engines, a silent empirical-Bayes fallback that masked a parameter-key mismatch, an information-criterion parameter-count error for categorical covariates, and an unguarded state-and-audit update in the workflow path — each of which was fixed and pinned with a regression test. The adversarial-review procedure is reported here as a method because the correctness of agentic tooling is not established by unit tests alone.

### Computational environment and reproducibility

The framework runs on Python 3.13 with numpy, scipy, and pandas; the external engine uses R 4.5.1 with nlmixr2. When an x86-64 Python process drives the pipeline on an Apple-silicon host, a child `Rscript` inherits the x86-64 preference and compiles an x86-64 model object that the native arm64 R cannot load, so the adapter runs R under `arch -arm64`. The condition is detected by probing whether an arm64 binary can execute — a check that does not depend on `sysctl.proc_translated`, which a restricted environment can block — and the fix is a no-op on native arm64, Intel, and non-macOS hosts. This detail is documented because, when the check is bypassed (for example in a sandbox that blocks the probe), the external fit fails with an architecture-mismatch error and that engine is reported as failed rather than contributing a result; the demonstration output is therefore pinned to a captured run. The demonstration and the framework test suite each run from a single command.

### Evaluation data

The framework was evaluated on simulated concentration–time data, not on clinical measurements. One-compartment oral profiles were generated for the primary demonstration from typical parameters (clearance 5 L/h, central volume 50 L, absorption rate constant 1 h⁻¹) at a 100 mg dose, with lognormal between-subject variability of approximately 22% applied to each structural parameter and proportional residual error of approximately 6%, sampled at 0.5, 1, 2, 3, 4, 6, 8, 12, and 24 h for six subjects under a fixed random seed so that every reported figure is reproducible. A two-compartment oral scenario was used to demonstrate graceful handling of a structure the external engine does not support. The use of simulated data is a deliberate scope choice for a framework description and is revisited as a limitation in the Discussion.
